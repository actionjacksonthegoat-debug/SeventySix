# Certificate Lifecycle

All certificates and expirable credentials in the SeventySix system.

## Certificate Inventory

| Certificate | Location | Lifetime | Rotation |
|---|---|---|---|
| Dev SSL (Angular) | `SeventySix.Client/ssl/` | 730 days | `scripts/generate-dev-ssl-cert.ps1` |
| Data Protection | `SeventySix.Server/SeventySix.Api/keys/` | Per script | `scripts/generate-dataprotection-cert.ps1` |
| Internal CA root | `docker/certs/ca.crt` | 10 years | `scripts/generate-internal-ca.ps1` |
| Internal service certs | `docker/certs/<service>/` | 1 year | `scripts/generate-internal-ca.ps1` |
| Cloudflare origin cert | Cloudflare dashboard | 15 years | Cloudflare dashboard renewal |
| Let's Encrypt (Caddy) | Auto-managed by Caddy | 90 days | Automatic (ACME at ~60 days) |

## Internal CA Certificates

Generated by `scripts/generate-internal-ca.ps1`. All stored in `docker/certs/` (gitignored).

**Server certificates** (present TLS identity):
- `postgres/` — PostgreSQL server
- `valkey/` — Valkey cache server
- `otel-collector/` — OpenTelemetry Collector
- `jaeger/` — Jaeger tracing
- `prometheus/` — Prometheus metrics
- `grafana/` — Grafana dashboards

**Client certificates** (authenticate to servers):
- `api/` — .NET API connecting to PostgreSQL, Valkey, OTEL Collector
- `otel-client/` — OTEL Collector connecting to Jaeger
- `nginx-proxy/` — nginx connecting to observability services

## mTLS Coverage

| Connection | Encryption | Auth |
|---|---|---|
| API → PostgreSQL | TLS (VerifyFull) | Client cert + password (scram-sha-256) |
| API → Valkey | TLS (CA verified) | Client cert + password |
| API → OTEL Collector | TLS (gRPC) | Client cert |
| OTEL Collector → Jaeger | TLS | Client cert |
| Caddy → API | HTTP | localhost-only (no network exposure) |
| Caddy → Client nginx | HTTP | localhost-only (no network exposure) |
| Client nginx → Grafana/Jaeger/Prometheus | HTTP | Docker bridge only, auth_request protected |

## File Permission Model

Private keys use `600` base permissions with POSIX ACLs (`setfacl`) to grant read access to exactly the container UID that mounts each key. No key is world-readable.

| Key file | Base | ACL grants |
|---|---|---|
| `ca.key` | `600 deploy:deploy` | None — never mounted in containers |
| `postgres/server.key` | `600 70:70` | None — PostgreSQL validates strict ownership |
| `otel-collector/server.key` | `600 deploy:deploy` | `u:10001:r` (otel-collector image UID) |
| `otel-client/client.key` | `600 deploy:deploy` | `u:10001:r` (otel-collector image UID) |
| `jaeger/server.key` | `600 deploy:deploy` | `u:10001:r` (jaeger image UID) |
| `api/client.key` | `600 deploy:deploy` | `u:999:r` (appuser), `u:59000:r` (redis-exporter) |
| `valkey/server.key` | `600 deploy:deploy` | None — Valkey runs as root (UID 0) |
| `nginx-proxy/client.key` | `600 deploy:deploy` | None — nginx runs as root (UID 0) |

The `generate-internal-ca.ps1` script applies these ACLs automatically on Linux when `setfacl` is available.

## Rotation Procedures

### Service Certificates (Annual)

On the production server (Linux, no PowerShell required):
```bash
# Clone or update the repo to get the latest scripts
cd /srv/seventysix

# Remove expired service certs (preserve CA so CA rotation isn't triggered)
find docker/certs -name '*.crt' ! -name 'ca.crt' -delete
find docker/certs -name '*.key' ! -name 'ca.key' -delete

# Re-generate all service certs + apply correct ACLs
pwsh scripts/generate-internal-ca.ps1

# Restart affected services to pick up new certs
docker compose -f docker-compose.production.yml restart
```

### Internal CA Root (10-Year)

The root CA rarely needs rotation. When it does:

1. Delete `docker/certs/ca.crt` and `docker/certs/ca.key`
2. Run `pwsh scripts/generate-internal-ca.ps1 -Force` — regenerates CA + all service certs + ACLs
3. Restart all services

### Emergency Rotation (Compromised Cert)

1. Delete `docker/certs/` directory
2. Run `pwsh scripts/generate-internal-ca.ps1` — regenerates CA + all certs + ACLs
3. Restart all services: `docker compose -f docker-compose.production.yml down && docker compose -f docker-compose.production.yml up -d`

## Bootstrap Integration

The `scripts/bootstrap.ps1` script generates all internal CA certificates as part of Phase 4 (Certificate Generation). Certificates are created on first run and skipped if already present.

## Design Decisions

**Localhost traffic remains HTTP**: Caddy → API and Caddy → Client nginx communicate over `127.0.0.1`. TLS on localhost provides no security benefit — an attacker with access to loopback traffic already has root access to the host.

**Observability dashboard traffic remains HTTP**: Grafana, Jaeger, and Prometheus have no external port bindings and are accessed only through the client nginx with `auth_request` admin verification. Adding TLS would require maintaining a separate production nginx.conf, with marginal security benefit for read-only dashboard traffic on an isolated Docker bridge.
